# Πάγωμα εξαρτήσεων του slacker για το επόμενο Slackware stable

Οδηγός για το πώς να κλειδώσεις τις εξαρτήσεις του slacker ώστε να χτίζει
αναπαραγώγιμα για όλη τη ζωή της επόμενης σταθερής έκδοσης του Slackware, και
γιατί ο σωστός τρόπος να βρεις το ελάχιστο rustc περνάει από το rustup.

## 1. Τι σπάει στ' αλήθεια, και τι όχι

Ο δικός σου κώδικας δεν σπάει ποτέ από μόνος του. Η Rust δίνει εγγύηση
συμβατότητας προς τα πίσω: ένας νεότερος rustc χτίζει τον παλιό σου κώδικα χωρίς
αλλαγές. Άρα ο κίνδυνος δεν είναι ποτέ ο νεότερος compiler.

Ο κίνδυνος είναι ένας και μόνο: τα ανοιχτά εύρη εκδόσεων στο Cargo.toml. Γραμμές
όπως `regex = "1"` ή `clap = "4"` σημαίνουν "πάρε ό,τι νεότερο ταιριάζει στο
εύρος". Κάποια στιγμή στο μέλλον μια νέα έκδοση ενός crate θα κάνει ένα από τα
δύο:

- θα ανεβάσει το δικό της ελάχιστο rustc (MSRV) πάνω από αυτόν που έχεις, ή
- θα αλλάξει το API της.

Σε όποια από τις δύο περιπτώσεις, ένα build που τραβάει "το νεότερο" θα αποτύχει.
Όλη η διαδικασία παγώματος υπάρχει για να κλείσει αυτή τη μία τρύπα.

Η λύση είναι τρία πράγματα που δουλεύουν μαζί:

1. Pin: κλείδωμα των ακριβών εκδόσεων μέσω του Cargo.lock.
2. Vendor: αντιγραφή του πηγαίου κώδικα των εξαρτήσεων τοπικά, ώστε το build να
   είναι offline και αυτάρκες.
3. Declare: δήλωση του ελάχιστου rustc μέσω του πεδίου rust-version, ως δίχτυ
   ασφαλείας.

## 2. Πότε παγώνεις

Όχι "όταν βγει το νέο stable" αόριστα, αλλά τη στιγμή που οριστικοποιείς μια
έκδοση του slacker απέναντι στον rustc που φέρνει το νέο stable. Εκείνο το ζευγάρι
(συγκεκριμένος rustc μαζί με συγκεκριμένες εκδόσεις crates) είναι ένα γνωστό-καλό
snapshot. Από εκεί και πέρα, νεότερος rustc χτίζει μια χαρά, και τα νεότερα crates
δεν τραβιούνται γιατί το Cargo.lock τα μπλοκάρει.

Προς το παρόν το stable είναι πολύ πίσω για να χτίσει το slacker. Όταν βγει η νέα
σταθερή έκδοση βασισμένη στο current που τρέχεις τώρα, τότε εκτελείς τα παρακάτω.

## 3. Τα "επικίνδυνα" αρχεία

Cargo.toml: εδώ είναι η επιφάνεια κινδύνου, δηλαδή τα ανοιχτά εύρη. Δεν χρειάζεται
να τα πειράξεις. Τα δαμάζει το Cargo.lock.

Cargo.lock: αυτό είναι το αντίδοτο. Το slacker είναι εφαρμογή (binary), όχι
βιβλιοθήκη, άρα αυτό το αρχείο το βάζεις στο git (commit). Όταν υπάρχει, κάθε
build χρησιμοποιεί ακριβώς αυτές τις εκδόσεις, ποτέ "το νεότερο". Σιγουρέψου ότι
δεν το αποκλείει το .gitignore.

## 4. Τα "επικίνδυνα" crates

Δεν χρειάζεται να κάνεις κάτι χειροκίνητα ανά crate, γιατί το Cargo.lock τα πιάνει
όλα μαζί. Είναι όμως καλό να ξέρεις πού να κοιτάς αν κάποτε ξεπαγώσεις και κάτι
σπάσει:

- Το TLS stack: native-tls, και από κάτω openssl-sys και openssl. Έχουν build
  scripts και εξαρτώνται από το openssl του συστήματος. Είναι το πιο συχνό σημείο
  σπασίματος σε builds διανομών.
- Οι βάσεις των proc-macros: syn, proc-macro2, quote. Έρχονται transitively (μέσω
  του clap) και ιστορικά ανεβάζουν το πάτωμα του MSRV.
- clap (φέρνει transitively το clap_lex 1.1.0, που είναι γραμμένο σε Rust edition
  2024 και άρα ανεβάζει το πάτωμα στο 1.85), το regex stack (regex-automata,
  regex-syntax), και χαμηλού επιπέδου crates όπως libc, cc, once_cell.

## 5. Η σωστή λύση για το ελάχιστο rustc: rustup

Για να ξέρεις σε ποια έκδοση rustc πρέπει να δηλώσεις ότι χτίζει το slacker, η
σωστή λύση είναι το rustup. Το έχεις ήδη εγκατεστημένο (rustup-1.29.0 από
conraid), οπότε δεν χρειάζεται κανένα ψεύτικο στήσιμο.

Γιατί rustup και όχι μάντεμα: το εργαλείο cargo-msrv κάνει bisection πάνω σε
πραγματικές εκδόσεις rustc, εγκαθιστώντας τες και δοκιμάζοντας αν χτίζει με την
κάθε μία. Αυτό απαιτεί rustup, γιατί μέσω rustup κατεβαίνουν και τρέχουν οι
διάφορες παλιές toolchains. Έτσι παίρνεις το ακριβές πάτωμα, μαζί με το ποιο crate
ή ποια γραμμή κώδικα το ορίζει, αντί να υποθέτεις.

Εκτέλεση:

```
cargo install cargo-msrv
cd /mnt/data/GITHUB/slacker
cargo msrv find
```

Το αποτέλεσμα είναι το ελάχιστο rustc που χτίζει το slacker. Στη μέτρηση που έγινε
βγήκε 1.85.1. Προσοχή: το πάτωμα δεν το ορίζει η έκδοση του clap αυτή καθαυτή,
αλλά το ότι μια εξάρτηση (το clap_lex 1.1.0, που έρχεται μέσω του clap) είναι
γραμμένη σε Rust edition 2024, η οποία σταθεροποιήθηκε στο Rust 1.85. Κάθε rustc
παλιότερος του 1.85 αποτυγχάνει με μήνυμα "feature edition2024 is required". Αυτό
το νούμερο, το 1.85.1, το βάζεις στο rust-version (βλέπε ενότητα 7, βήμα 3).

### Προσοχή: δύο toolchains στο ίδιο μηχάνημα

Το rustup έχει τη δική του toolchain, που την κατεβάζει το ίδιο, και τρέχει
παράλληλα με το system rust 1.96 του conraid. Δηλαδή στο μηχάνημα μπορεί να
συνυπάρχουν δύο rustc. Για το cargo-msrv αυτό δεν είναι πρόβλημα, αντιθέτως το θες
ακριβώς γι' αυτό, να δοκιμάζει πολλές εκδόσεις.

Όμως το release build που θα πακετάρεις πρέπει να το χτίσεις με τον rustc που θα
φέρει το νέο stable, όχι με μια τυχαία rustup toolchain. Αλλιώς το MSRV που
μέτρησες δεν αντιστοιχεί στο πραγματικό target. Έλεγξε ποια toolchain είναι ενεργή
πριν πακετάρεις:

```
rustup show
rustup which cargo
which -a cargo rustc
```

Το πρώτο δείχνει ποιες toolchains είναι εγκατεστημένες και ποια είναι default. Το
δεύτερο δείχνει ποιο cargo θα τρέξει όντως μέσω rustup. Το τρίτο δείχνει τα
μονοπάτια για system (conraid) και rustup ξεχωριστά.

## 6. Το MSRV ως δίχτυ ασφαλείας

Αφού έχεις το νούμερο από το cargo-msrv, δήλωσέ το στο Cargo.toml:

```
[package]
rust-version = "1.85.1"
```

Με αυτό ενεργοποιείται ο MSRV-aware resolver του cargo (τον έχεις στο 1.96): ένα
μελλοντικό cargo update θα αρνείται να τραβήξει crate που ξεπερνά αυτό το πάτωμα.
Το Cargo.lock παραμένει η σκληρή εγγύηση. Το rust-version είναι η πρόσθετη
προστασία ώστε να μη σου σπάσει σιωπηλά το πάτωμα σε μελλοντικό update.

## 7. Η διαδικασία παγώματος βήμα προς βήμα

Εκτέλεσέ την όταν βγει το νέο stable, στο μηχάνημα build, με τον rustc του stable
ενεργό.

Βήμα 1. Toolchain στόχος. Επιβεβαίωσε ότι χτίζεις με τον rustc που φέρνει το νέο
stable:

```
rustc --version
```

Αυτό είναι το πάτωμά σου.

Βήμα 2. Build και test καθαρά, με τον δικό σου πραγματικό Cargo.toml (όχι κανένα
πειραγμένο για παλιότερο rustc):

```
cargo build --release
cargo test
```

Αν θες την πιο φρέσκια συμβατή συλλογή εξαρτήσεων, τρέξε cargo update και ξανα-test.
Ό,τι βγει πράσινο εδώ είναι το snapshot σου.

Βήμα 3. Δήλωσε το MSRV στο [package] του Cargo.toml με το νούμερο που σου έδωσε το
cargo msrv find:

```
rust-version = "1.85.1"
```

Βήμα 4. Κλείδωσε και επιβεβαίωσε ότι το lock είναι αυθεντία:

```
cargo build --release --locked
git add -f Cargo.lock
```

Το --locked αποτυγχάνει αν το lock δεν ταιριάζει, οπότε σου εγγυάται ότι χτίζεις
ακριβώς ό,τι κλείδωσες.

Βήμα 5. Vendor τις πηγές των εξαρτήσεων:

```
cargo vendor vendor/
```

Η εντολή τυπώνει στο stdout ένα block ρυθμίσεων. Σώσ' το στο .cargo/config.toml:

```
[source.crates-io]
replace-with = "vendored-sources"

[source.vendored-sources]
directory = "vendor"
```

Βήμα 6. Επιβεβαίωσε ότι χτίζει εντελώς offline. Αν θες, κόψε και το δίκτυο για
σιγουριά:

```
cargo build --release --offline --locked
```

Συντομία: το --frozen ισοδυναμεί με --locked και --offline μαζί.

Βήμα 7. Πακετάρισμα για διανομή. Δύο tarballs, δεμένα στο git tag της έκδοσης:

- slacker-X.Y.Z.tar.xz με τον πηγαίο κώδικα, το Cargo.lock και το
  .cargo/config.toml.
- slacker-X.Y.Z-vendor.tar.xz με τον φάκελο vendor. Είναι μεγάλος, αυτό είναι το
  τίμημα ενός αυτάρκους build.

Βήμα 8. SlackBuild. Ξεπακετάρει και τα δύο δίπλα-δίπλα και χτίζει χωρίς δίκτυο:

```
cargo build --release --offline --locked
```

Έτσι το build είναι αναπαραγώγιμο για πάντα.

## 8. Κόλπα και παγίδες

Μην κάνεις vendor το openssl με το vendored feature του. Σε πακέτο διανομής θες να
γίνεται link με το openssl του συστήματος, ώστε τα security updates του Slackware
να φτάνουν αυτόματα και στο slacker. Ο πηγαίος κώδικας του openssl-sys μπαίνει
κανονικά στο vendor (ως -sys crate), αλλά στο build γίνεται link με τη βιβλιοθήκη
του συστήματος. Αυτό είναι το σωστό.

Η φορά του κινδύνου είναι μονόδρομη. Μόνο τα νεότερα crates σε απειλούν, ποτέ ο
νεότερος rustc. Άρα lock μαζί με vendor σημαίνει ότι το slacker χτίζει με την ίδια
ή νεότερη toolchain επ' άπειρον.

Το ξεπάγωμα είναι συνειδητή ενέργεια, όχι κάτι τυχαίο. Όταν θες security updates
στα crates ή πας σε επόμενο stable: cargo update, μετά ξανά cargo vendor, μετά
re-test, και ανεβάζεις την έκδοση του slacker.

Ένα snapshot ανά release. Δέσε το Cargo.lock και τον φάκελο vendor σε κάθε git
tag. Κάθε εκδοθείσα έκδοση slacker έχει το δικό της αναπαραγώγιμο σύνολο
εξαρτήσεων.

Το παγωμένο Cargo.toml που παραδίδεις στη διανομή είναι ο δικός σου, στον rustc
του stable. Όχι κανένα πειραγμένο Cargo.toml για παλιότερο rustc.

## 9. Σύνοψη σε μία γραμμή

Βρες το ελάχιστο rustc με rustup και cargo-msrv, δήλωσέ το στο rust-version, κάνε
commit το Cargo.lock, και διένειμε vendored εξαρτήσεις. Τότε το slacker θα χτίζει
αναπαραγώγιμα όσο ζει το Slackware-στόχος, ανεξάρτητα από το τι κάνει το crates.io
ή πόσο ανεβαίνουν τα MSRV των crates στο μέλλον.
