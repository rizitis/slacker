================================================================================
slacker — source code map
================================================================================

A guide to src/ for anyone forking, fixing, or extending slacker. For each
module: what it owns, the key items, and where to look first for a given change.
Read this top-to-bottom once; after that use it as an index.

The mental model: config.rs reads your /etc/slacker files -> repo.rs fetches and
parses each repo's metadata into AvailPkg records -> pkgdb.rs unifies them into
one priority-ordered database and answers "what wins / what upgrades / what
matches" -> main.rs turns a CLI command into a plan and calls system.rs to run
the real pkgtools. download.rs, gpg.rs, manifest.rs, mirrors.rs, changelog.rs,
history.rs, template.rs, newconfig.rs are focused helpers used along the way, and
dist.rs holds the release algebra that main.rs's upgrade-dist drives.

Everything is synchronous. Errors are String (Result<_, String>) on purpose —
simple, greppable messages, no error crates.

--------------------------------------------------------------------------------
pkg.rs            (~192 lines)   Slackware package-name parsing
--------------------------------------------------------------------------------
Owns the PkgId type: a parsed "name-version-arch-build" 4-tuple, the atom the
whole program passes around.

  PkgId            struct { name, version, arch, build }
  PkgId::parse     split a filename/dirname into the 4 fields
  PkgId::tag()     full "name-version-arch-build" string
  PkgId::build_tag()  the build field minus leading digits
                      (1_SBo -> _SBo, 7cf -> cf, 1 -> "")

WHY build_tag MATTERS: it is how slacker tells "which source built this" without
a database. The whole priority-protection model keys off it. If you change how
tags are derived, expect upgrade-all and the @_tag selector to shift.

LOOK HERE IF: package names parse wrong (epoch, odd arch like fw/x86, names with
dashes). Parsing is positional from the right, so unusual names are the risk.

--------------------------------------------------------------------------------
config.rs         (~1080 lines)  read & model /etc/slacker
--------------------------------------------------------------------------------
Parses the plain-text config files into typed structs. No network, no side
effects beyond reading files.

  Config           struct: arch, cache_dir, pkg_db_dir, adm_dir, resolve_deps,
                   ignore_tags, repos, tag_priorities, blacklist
                   (Vec<BlacklistRule>), config_dir, verify, distro_upgrade_mirror,
                   + helpers (repo_by_name, repos_by_priority, blacklist_hit,
                   official_repo_name, is_ignored_tag)
  distro_upgrade_mirror  Option<String> from distro-upgrade.conf key
                   DISTRO_UPGRADE_MIRROR (parse_distro_upgrade_mirror: trim, drop
                   a trailing /, None when empty). A local source (file:// tree,
                   http clone, mounted ISO) that upgrade-dist re-points to instead
                   of the network; main.rs validates target + reachability.
  ADM_DIR/PKG_DB_DIR  ADM_DIR (slacker.conf, default /var/adm) is the pkgtools
                   admin root; pkg_db_dir is an explicit PKG_DB_DIR when set
                   (back-compat override), else adm_dir.join("packages"). adm_dir
                   itself is what history.rs walks (packages/ + removed_packages/).
  BlacklistRule    struct: optional @repo scope, optional series, optional
                   compiled regex (matched against the full package id). With
                   matches(id, series, repo) + repo()/pattern()/describe()
  parse_blacklist_rule  parse one `[@repo] REGEX|series/` line (pub: the
                   `frozen` command reuses it to validate user input)
  Repo             struct: priority, name, url, official, immutable, subtree,
                   verify (per-repo override) + verify_policy(global) resolver +
                   download_base()/join_download_url() (subtree -> parent/root
                   URL for packages & GPG-KEY; metadata still from the repo URL)
  TagPriority      struct: priority, name, tag   (the `priority name tag` lines)
  Check            enum: Gpg | Md5 | Sha — one verification method
  VerifyPolicy     enum: All | Required(list) | None, with wants()/requires();
                   parsed from `VERIFY=` (slacker.conf) or `verify=` (repos)
  Config::load     the entry point; ties all files together
  detect_arch      reads ARCH from the installed aaa_base package
  parse_repos      splits binary-repo lines from tag-priority lines: the 3rd
                   field is a URL / "mirror" / "mirror/<subpath>" -> binary repo;
                   otherwise -> tag line. The url keyword resolves to the active
                   mirror (bare `mirror`) or the active mirror plus a subpath
                   (`mirror/extra`, `mirror/testing`). Also parses `official`,
                   `immutable`, `subtree` and `verify=` flags (any order; an
                   unknown flag is a fail-fast error)
  parse_mirrors    enforces "exactly one uncommented mirror"
  parse_distro_upgrade_mirror  the DISTRO_UPGRADE_MIRROR value parser (above)

KEY RULES ENFORCED HERE:
  - binary-repo priorities must be DISTINCT (duplicate -> fail-fast error)
  - tag-priority lines MAY share a priority
  - two-or-more active mirrors -> error; zero -> the official URL is empty

LOOK HERE IF: a config line is misread, you want a new slacker.conf KEY, a new
repos-line form, or to change arch detection. Add a KEY in Config::load's
key/val handling; add a struct field alongside.

--------------------------------------------------------------------------------
repo.rs           (~690 lines)   fetch + parse a repo's metadata
--------------------------------------------------------------------------------
Turns a Repo's remote files into AvailPkg records and keeps the cache fresh.

  AvailPkg         struct: id (PkgId), repo, filename, location/series, summary,
                   md5, sha, url helpers — one available package in one repo
  update_repo      download PACKAGES.TXT (required) + CHECKSUMS.md5 / .asc /
                   .sha256 / ChangeLog (best-effort) into CACHE_DIR/repos/<name>/
  ensure_manifest  lazily fetch+decompress MANIFEST.bz2 on first file-search
  load_repo        read cached metadata -> Vec<AvailPkg>, arch-filtered
  parse_packages_txt / parse_checksums   the actual format parsers
  arch_compatible  keep native arch + noarch + fw (firmware) + x86 (32-bit
                   headers); this mirrors slackpkg, do not narrow casually
  read_text_lossy  pub(crate): UTF-8-lossy reads — some repos (extras) have
                   non-UTF-8 bytes; use this for every metadata read
  fetch_dep / parse_dep   pull and parse a package's `.dep` file (slackpkg+)
  previous_names          the .prev snapshot install-new diffs against (by
                          package NAME, so a version/build bump is not "new")
  fetch_changelog_text    on-demand ChangeLog fetch for show-changelog
  invalidate_metadata     drop a repo's cached PACKAGES.TXT/CHECKSUMS after a
                          failed GPG verification, so unverified data is unused
  quarantine_dir / trusted_dir / *_marker   cache markers under CACHE_DIR:
                          quarantine/<name> (frozen, inert) | trusted/<name>
                          (vetted-pass or user-trusted) | neither (untrusted ->
                          light-vetted on next update). is_quarantined /
                          is_hard_quarantined / mark/clear helpers back the
                          vet-repo / trust-repo / distrust-repo commands; load
                          skips a quarantined repo entirely

LOOK HERE IF: a repo's PACKAGES.TXT/CHECKSUMS parses wrong, a package is wrongly
filtered out by arch, MANIFEST handling misbehaves, or .dep contents are
misread. bzip2 (MANIFEST) and gpg are shelled out, not Rust deps.

GOTCHA: always route new metadata reads through read_text_lossy, or a single
non-UTF-8 byte will crash a command (this was a real bug on the extras repo).

--------------------------------------------------------------------------------
pkgdb.rs          (~844 lines)   the unified, priority-ordered brain
--------------------------------------------------------------------------------
Merges every repo's AvailPkg list into one DB and answers the hard questions.
This is where most behavioural bugs live — read carefully before changing.

  PkgDb            struct: all (Vec<AvailPkg>) + priority map (name->i32)
  PkgDb::load      build from Config (calls repo::load_repo per repo)
  search           EXACT package name, case-insensitive (one row per name,
                   highest-priority repo winning); info/file-search do substrings
  resolve          one name (or repo:name pin) -> the winning candidate
  match_pattern    THE pattern resolver. Order matters:
                     1. @repo / @_tag  -> set selector (returns early)
                     2. series name    -> that series only (no substring)
                     3. exact name / substring
                   pinning (repo:name) is handled via split_pin
  installed_priority   given an installed PkgId, decide its "owning" priority:
                     1. user tag-priority line for its build_tag
                     2. else highest-priority binary repo that ships that tag
                        (cf->conraid, alien->alienbob auto-detected)
                     3. else official priority for an empty tag
                     4. else i32::MAX (unknown -> protect, never touch)
  upgrades_for     produce the upgrade-all plan: a candidate is proposed only
                   from a repo of HIGHER or EQUAL priority; equal-priority needs
                   a genuine revision change; lower priority is skipped (this is
                   what prevents cross-repo migration and downgrades)
  upgrade_respects_priority / installed_outranks   the same rule as two
                   booleans, reused by the upgrade/reinstall/@selector targeting
                   and the dependency resolver (candidate >= installed source ->
                   may replace; installed >= candidate -> keep installed)
  series_of / repo_for_tag   look up an installed package's series and source
                   repo (used to evaluate `series/` and `@repo` blacklist rules
                   against installed packages, which carry no series themselves)
  newly_added      install-new: names present now but absent from .prev and not
                   installed
  names_provided_by  clean-system baseline: the set of package names provided
                   by a given set of repos (official + every immutable repo),
                   or all repos when no baseline is given. has_repo_packages
                   reports whether a repo's metadata is loaded (the safety guard
                   that makes clean-system refuse to run on missing baseline data)
  repo_for_tag     which repo owns a build tag (first AvailPkg carrying it) ->
                   used by clean-system's immutable rule and installed_attribution
  orphans          installed names found in NO repo (now used by
                   generate-template; clean-system uses the baseline rule above,
                   not this)
  is_repo / tag_in_use / repo_build_tags / all_repos / all_build_tags
                   helpers for @-selectors and the "did you mean" diagnostics

LOOK HERE IF: upgrade-all proposes something wrong (migration, downgrade, a
missed upgrade) -> upgrades_for + installed_priority. A pattern matches too much
or too little -> match_pattern. clean-system lists the wrong things ->
names_provided_by + the baseline/immutable/IGNORE_TAGS/blacklist logic in
cmd_clean_system.

THE SUBTLEST CODE IN THE PROJECT is installed_priority + upgrades_for. The
"distinct, high priority locks a repo's packages" guarantee depends on them.
There are unit tests pinning this (sbo_not_migrated, no_downgrade,
genuine_self_upgrade, higher_priority_same_version) — keep them green.

--------------------------------------------------------------------------------
download.rs       (~380 lines)   bytes over http/https/file
--------------------------------------------------------------------------------
The only module that talks to the network (via ureq + rustls).

  get_bytes        fetch a URL into memory (used for keys, .dep, small files)
  download_to      stream a URL to a destination path
  md5_file         md5 of a local file (md-5 crate) for CHECKSUMS verification
  sha256_file      sha-256 of a local file, via the `sha256sum` command (no
                   hashing crate; shelled out like gpg/bzip2)
  file_url_to_path + percent_decode   file:// support, reads the FS directly

LOOK HERE IF: TLS/proxy/timeout issues, file:// URLs, or md5 mismatches. TLS is
rustls with bundled roots (no system OpenSSL/libssl), so there is no libssl
dependency to satisfy on an old base; a CA/proxy problem surfaces here.

--------------------------------------------------------------------------------
system.rs         (~212 lines)    the bridge to real pkgtools
--------------------------------------------------------------------------------
Reads the installed-package DB and shells out to the Slackware tools. This is
the only module that mutates the system.

  installed_packages   parse PKG_DB_DIR (default ADM_DIR/packages, i.e.
                   /var/adm/packages)
  is_installed / installed_by_name   lookups over that list
  install / upgrade_only / reinstall / remove_package
                   thin wrappers over installpkg/upgradepkg/removepkg
  upgrade_install_new  upgradepkg --install-new (install-or-upgrade in one), used
                   on the distribution-upgrade path
  cached_pkg_path  CACHE_DIR/packages/<repo>/<filename> — the download target
  version_codename / version_id   read /etc/os-release (VERSION_CODENAME /
                   VERSION_ID): the running release, for revert-pkg's -current
                   guard and the release-mismatch / upgrade-dist logic

LOOK HERE IF: install/upgrade/remove behaves differently from the native tools,
or the installed-DB parse is wrong. slacker is intentionally a THIN layer here:
it does not reimplement pkgtools, it calls them. Keep it that way.

--------------------------------------------------------------------------------
dist.rs           (~304 lines)   distribution-upgrade engine (release logic)
--------------------------------------------------------------------------------
The pure release-reasoning core behind `upgrade-dist` and the release-mismatch
guard. No side effects: it parses and decides, main.rs acts.

  Release          enum: Current | Stable(String)  (a Slackware release)
  Route            enum: the kind of distribution upgrade (stable->current, etc.)
  show             human label for a Release ("-current", "15.0", ...)
  parse_release_from_os   read a Release from os-release (VERSION_ID/CODENAME):
                   codename=current -> Current, else trim a stray '+' off the
                   VERSION_ID -> Stable. The running side.
  parse_release_from_url  read the release a mirror/repo URL targets from its
                   slackware{arch}-<suffix> segment (None when absent)
  release_suffix / parse_target   the suffix string for a Release, and parse the
                   upgrade-dist argument ("current" | "15.1" -> Release)
  dist_route       THE GATE: a fail-closed whitelist of allowed routes
                   (15.0 -> current, 15.0 -> a newer stable); anything else Err.
                   Decides only DIRECTION; the URL transform is identical for all
                   allowed routes.
LOOK HERE IF: a distribution-upgrade route is wrongly allowed/refused
(dist_route), or a release is misread from os-release/a URL. The actual mirror/
repo rewriting, the plan build, batching and execution live in main.rs
(cmd_upgrade_dist + the dist_* family); this module is just the release algebra.
NOTE: repo_release_token in main.rs ADDS a bare current/X.Y path-segment fallback
on top of parse_release_from_url for the release-mismatch guard (third-party
repos like alienbob's sbrepos/current that lack the slackware{arch}- prefix).

--------------------------------------------------------------------------------
history.rs        (~420 lines)   package-change timeline from the admin dirs
--------------------------------------------------------------------------------
Reconstructs a chronological log of every package change on the box from the
pkgtools admin dirs under ADM_DIR — so it sees changes from any tool (slacker,
slackpkg, sbopkg, installpkg/upgradepkg/removepkg), not just slacker's own. Pure
logic, unit-tested, no network. Shells out to nothing (uses std MetadataExt for
mtime/ctime).

  collect(adm_dir)     the entry point -> Timeline { events, clock, current }.
                       Reads packages/ (current installs; mtime = install time)
                       and removed_packages/ (departures).
  RawRecord            one admin-dir entry: pkg (PkgId), installed_at (mtime),
                       departure: Removed(ctime) | Upgraded(ctime) | None.
                       A plain removed_packages/<id> = removepkg (ctime = removal,
                       mtime preserved = its install). An <id>-upgraded-<TS> =
                       the OLD version replaced by upgradepkg.
  build_events         PURE core: per-NAME chronological chaining. Each name's
                       records are successive tenures sorted by install time; an
                       Upgraded-away tenure pairs with the very next tenure (within
                       ~2 days = real in-place successor, consumed). If that
                       successor's record was LOST to a removed_packages name
                       collision, the target VERSION is inferred from the next
                       known tenure (display only, not consumed) — so an upgrade
                       reads "old -> new", never "old -> ?". Reinstall = the same
                       id departed before this install.
  Event / EventKind    Event { when, pkg, kind }; kind = Installed{reinstall} |
                       Removed | Upgraded{ to: Option<PkgId> }
  LocalClock           self-calibrating UTC->local offset from (upgraded-filename
                       local time, that file's ctime) pairs, nearest sample per
                       event (DST-correct, no tz crate; falls back to UTC). format
                       / local_date for display + --since filtering.
  days_from_civil / civil_from_days   Hinnant date math (no chrono).

WHY removed_packages IS LOSSY: plain removepkg records collide on the package id
(a later removal overwrites an earlier one); only -upgraded-<timestamp> records
are unique. That is the structural reason build_events sometimes has to infer an
upgrade target instead of reading it.

LOOK HERE IF: a date is off (clock calibration), an upgrade shows the wrong target
or "?", or a reinstall is misclassified. The render (colour, columns, the
"reinstalled" same-version case) and the single-package source attribution live in
main.rs (render_history / source_of), not here.

manifest.rs       (~111 lines)   file-search
--------------------------------------------------------------------------------
  FileHit          struct: repo, package, path
  file_search      search the decompressed MANIFEST for a file/basename
LOOK HERE IF: file-search misses or over-matches. The MANIFEST is large and
fetched lazily by repo::ensure_manifest; this module only searches it.

--------------------------------------------------------------------------------
mirrors.rs        (~231 lines)   find-mirror: probe + rank official mirrors
--------------------------------------------------------------------------------
The engine behind the `find-mirror` command. Pure-ish network helper: fetch the
official mirror list, time each one, rank.

  MirrorResult     struct: url, latency, the mirror's own timestamp, freshness
  TOP_N            how many ranked mirrors to show (7)
  fetch_https_mirrors  scrape the official https mirror list (scheme-based parse,
                   resilient to page-format drift)
  probe_all        one timed HTTP range request per mirror for
                   slackware64-current/PACKAGES.TXT — measures real latency AND
                   reads the mirror's timestamp in the same request
  rank             drop mirrors stale by >48h vs the upstream master, sort the
                   reachable/fresh ones by speed, return the top N
LOOK HERE IF: find-mirror suggests a bad mirror or the mirror-list scrape breaks
(the official page changing layout is the known fragility). The command wiring
and the "make this line active in mirrors" hint live in main.rs (cmd_find_mirror);
this module only fetches, probes and ranks — it never edits the mirrors file.

--------------------------------------------------------------------------------
changelog.rs      (~77 lines)    check-updates / show-changelog
--------------------------------------------------------------------------------
  changelog_repo       which repo's ChangeLog to track (the official one)
  cached_changelog     read a repo's cached ChangeLog, if present
  check_repo_updates   official repo: compare cached vs remote ChangeLog;
                       every other repo: compare cached vs remote CHECKSUMS.md5
                       (smaller than PACKAGES.TXT, changes on any package change)
                       -> UpdateStatus
  UpdateStatus         enum used for the exit-100 "pending" signal
LOOK HERE IF: check-updates gives the wrong pending/exit status. show-changelog
takes an optional repo name; the on-demand fetch for an uncached repo lives in
repo::fetch_changelog_text.

--------------------------------------------------------------------------------
gpg.rs            (~280 lines)   key import + signature verify (fail-closed)
--------------------------------------------------------------------------------
  import_key           `update gpg`: fetch GPG-KEY (from the repo's download
                       base -> the root for a subtree repo) + gpg --import into
                       a per-cache keyring, then PIN its fingerprint on first
                       contact (TOFU). A later key whose fingerprint differs from
                       the pin is refused as a possible key-substitution attack.
                       Returns ImportOutcome / ImportError(KeyChanged | Other)
  read_pin / pin       per-repo fingerprint pin under cache/gpg/<name>.fpr
  verify_checksums     verify CHECKSUMS.md5.asc against the pinned key; FAIL-CLOSED
  verify_detached      verify any file against a detached .asc (per-package
                       .txz.asc at install); same pinned-key, fail-closed contract
  verify_against_pin   the shared check: GOODSIG + the signer must be the PINNED
                       key, else Tampered (key substitution) / Unverifiable
  Verify               enum result: NoSignature | Good(signer) | Tampered(msg) |
                       Unverifiable(msg)
  set_mode_700         lock down the keyring dir (unix)
LOOK HERE IF: GPG verification rejects a good repo or you need different key
handling. Note: this is intentionally strict — do not make verification
best-effort without a very good reason. gpg is shelled out.

--------------------------------------------------------------------------------
main.rs           (~8381 lines)  CLI, planning, and every command
--------------------------------------------------------------------------------
The orchestrator. clap definitions, the command dispatch, the install/upgrade
PLAN builder, and one cmd_* function per action.

  Cli / Cmd        clap structs/enums — the CLI surface. ADD A COMMAND HERE
                   (enum variant + match arm in run()); kebab-case is automatic.
  Outcome          exit-code enum (0 ok / 1 err / 20 none / 50 self / 100 pending)
  main             resets SIGPIPE to its default (restore_default_sigpipe, a raw
                   libc signal call, no extra crate) so piped/early-closed output
                   like `slacker ... | head` ends quietly instead of panicking
  run              dispatch a parsed Cmd to its cmd_* function
  confirm          the y/N prompt (honours --yes); choice_line renders the bracket
                   KEY letters in white for it and the new-config prompts

  PLAN BUILDING (the core install/upgrade machinery):
    InstallAction  Install / Upgrade / Reinstall
    PlanItem       one package + action + optional dep_for (parent) + optional
                   from (installed version, so upgrades show old -> new)
    expand_with_deps / add_with_deps   recursively pull .dep dependencies,
                     dedup, and attach dep_for so new deps can be shown up front
    ask_dep_conflict  the skip/replace/skip-all/abort prompt when an installed
                     dep from a LOWER-priority source differs from the repo
    resolve_protected_deps / ask_protected_dep   when an installed dep comes
                     from a HIGHER-or-equal-priority source it is kept by the
                     priority rule, but surfaced in a table with keep / replace /
                     keep-all (kept_detail formats the "installed X (pri) — repo
                     (pri) not applied" lines)
    PlanRow / print_table   render the coloured Action|Package|Version|Repo
                     plan tables (used by every plan-producing command —
                     install/upgrade/reinstall/remove/upgrade-all/install-new/
                     clean-system)
    print_plan / execute_plan   show the plan, then run it via system.rs.
                     execute_plan first calls enforce_release_match (the
                     release-mismatch guard, fail-closed unless --yes); the dist
                     path does NOT go through execute_plan, so it is exempt.
    fetch_and_verify  download + verify, gated by the repo's VerifyPolicy:
                     GPG (at update, in cmd_update) + per-package integrity here
                     (md5 and/or sha — at least one must pass; none available =>
                     stop). REFUSES to write through a pre-existing symlink.
    verify_unavailable_error   the "required method missing, here's how to relax
                     it" message (points at repos / slacker.conf)
    warn_unverified_repos   visible warning for repos whose policy does no checks
    report_pending_configs  after install/upgrade/upgrade-all, list any new *.new
                     files so the user can run `new-config`

  BLACKLIST HELPERS (thin wrappers over Config::blacklist_hit):
    bl_avail / bl_installed / bl_frozen   does a rule match an available
                     candidate / an installed package / either (freeze)? These
                     resolve the series + source repo a rule needs.
    frozen_warn      the `frozen` validator's soft-warning builder: unknown
                     @repo, or a regex containing whitespace (a package id never
                     does — a forgotten @ or quoting slip)

  SELECTORS / MATCHING HELPERS:
    collect          expand patterns -> winning packages (+ validates selectors)
    validate_selector  the @repo/@_tag check with the "did you mean" suggestion
    edit_distance / closest   Levenshtein for suggestions
    select_packages / select_packages_pkgid / parse_selection
                     the numbered multi-match chooser (Enter=all, ranges, n)

  COMMANDS: cmd_update, cmd_search, cmd_file_search, cmd_info, cmd_install,
  cmd_upgrade, cmd_reinstall, cmd_remove, cmd_download, cmd_upgrade_all,
  cmd_install_new, cmd_clean_system, cmd_clean_cache, cmd_new_config,
  cmd_check_updates, cmd_show_changelog, cmd_history, cmd_find_mirror,
  cmd_upgrade_dist, cmd_generate_template, cmd_install_template,
  cmd_remove_template, cmd_delete_template, cmd_frozen

  DISTRIBUTION UPGRADE (upgrade-dist) + RELEASE GUARD:
    cmd_upgrade_dist  the whole distribution-upgrade orchestration. Uses dist.rs
                      for the release algebra (running side from os-release,
                      target from the argument, dist_route whitelist gate), then:
                      dist_backup (Phase -1 escape-kit template) -> dist_transform
                      (re-point mirror + every mirror/subtree repo to the target,
                      comment out un-re-pointable third-party repos, blacklist ->
                      .bak + emptied; set_active_mirror_line when a local
                      DISTRO_UPGRADE_MIRROR is used, dist_rewrite_text otherwise)
                      -> dist_validate_local_mirror (segment==target / reachable,
                      before the point of no return) -> a single default-yes
                      confirm + silent update of all repos -> dist_upgrade_sets +
                      dist_install_new -> dist_execute.
    dist_upgrade_sets the priority-BYPASSING selection: for every installed
                      package take db.resolve's winner regardless of
                      priority/blacklist/frozen, split into (critical, rest).
                      DIST_CRITICAL (glibc-solibs, pkgtools, tar, xz, ...) is
                      ordered first; DIST_GPG_LAST (gnupg/gnupg2 + gpgme/libassuan/
                      libgpg-error/npth, the GnuPG verification chain) is deferred
                      to the END so package signatures keep verifying throughout.
    dist_install_new  the target's newly-added official packages not installed.
    dist_execute      download+verify, core all-first (abort if a core package
                      fails before anything is touched), then the rest in batches
                      that are deleted as they install (no whole-release disk
                      spike); a disk-space preflight stops early. Followed by
                      install-new, clean-system and a second confirming pass, then
                      the kernel/initrd + bootloader reminder.
    repo_release_token / system_release / enforce_release_match
                      the RELEASE-MISMATCH GUARD. system_release reads os-release;
                      repo_release_token reads a repo URL's release (dist::
                      parse_release_from_url, OR a bare current / clean X.Y
                      path-segment via is_release_version_segment, for third-party
                      repos like alienbob; None = release-agnostic, never flagged).
                      enforce_release_match (called at the top of execute_plan)
                      REFUSES a plan whose package comes from a repo built for a
                      different release than the system, unless --yes. status shows
                      the same mismatch via a "Repo release" row.

  FIND-MIRROR:
    cmd_find_mirror   wraps mirrors.rs (fetch_https_mirrors -> probe_all -> rank)
                      and prints the fastest fresh mirror as the line to make
                      active in the mirrors file. Never edits the file. Runs even
                      before any mirror is configured.

  REPO / TAG / STATUS / SAFETY COMMANDS:
    cmd_list_repos    repos with priority, verify, flags, and per-repo installed
                      counts (from installed_attribution)
    cmd_status        whole-setup health check (mirror, repos, keys, freshness,
                      reachability, pending) with "what to do next"
    cmd_add_repo / cmd_del_repo / cmd_add_tag / cmd_del_tag   edit the `repos`
                      file (validated via config::validate_repos_text, confirmed)
    cmd_vet_repo / cmd_trust_repo / cmd_distrust_repo   the quarantine model:
                      re-vet on demand, lift a quarantine (override), or freeze
    installed_attribution / attribute_tags   tag-based source mapping with a pure,
                      testable core attribute_tags(official, tag_priorities,
                      resolve_repo, installed) -> (per_repo, per_rule, per_other).
                      Precedence: official (empty tag) -> binary repo (repo_for_tag)
                      -> declared tag-rule name -> raw tag. There is NO "untracked"
                      bucket: an installed package's own build tag is always a
                      legitimate source. Feeds list-repos (the Inst column + the
                      "(declared, no installed package)" / "Installed under other
                      build tags" lines) and status's by-source view.
    cmd_history / render_history / source_of   the `history` command: read the
                      timeline via history::collect(adm_dir), filter
                      (--installed reads tl.current directly; --removed = Removed |
                      Upgraded; --upgraded = Upgraded; name exact-match; --since;
                      --last N), then render newest-first as a colour table.
                      source_of attributes a single PkgId (same precedence as
                      attribute_tags). render_history shows a same-version upgrade
                      as "reinstalled".
    cmd_clean_system  the baseline rule: foreign = not provided by official +
                      immutable repos; kept via blacklist / IGNORE_TAGS /
                      immutable attribution; refuses on missing baseline metadata
  page_output   pipe long output (history, show-changelog) through $PAGER /
                `less -FRX` when stdout is a TTY. The text is written from a scoped
                thread while the main thread waits on the pager, so a body larger
                than the pipe buffer cannot deadlock and `q` always quits cleanly;
                short output prints inline (less -F) and the terminal is left clean
                (less -X).

and self-contained; the shared complexity is in expand_with_deps/add_with_deps

================================================================================
General hints for a forker
================================================================================

- START by reading config.rs (what the tool consumes), then pkgdb.rs (how it
  decides), then the relevant cmd_* in main.rs. That path explains almost any
  behaviour.

- ADDING A COMMAND: add a Cmd variant + a match arm in run() + a cmd_* fn. If it
  takes patterns, reuse collect() so it gets @-selector validation and the
  multi-match chooser for free. Update four docs by hand: README.md,
  slacker.8, structure.md, CODE_MAP.txt (clap --help updates itself).

- THE PRIORITY MODEL is the heart and the easiest thing to break. Any change
  near installed_priority / upgrades_for (pkgdb.rs) should be checked against
  the upgrade tests. The invariant users rely on: a distinct high priority locks
  a repo's tagged packages against any lower repo, even a newer version.

- TAGS CARRY MEANING. build_tag() is load-bearing for upgrade protection,
  @_tag, IGNORE_TAGS, and clean-system. Treat it as a stable contract.

- KEEP IT A THIN LAYER. installpkg/upgradepkg/removepkg, gpg, and bzip2 are
  shelled out on purpose. Don't reimplement them in Rust; Slackware ships them.

- FAIL-CLOSED on GPG and on checksums. The verification model: GPG is checked at
  `update` (signs the CHECKSUMS file); per-package md5/sha is checked at install
  in fetch_and_verify. Policy lives in config.rs (VerifyPolicy/Check), parsed
  from VERIFY= (global) and verify= (per-repo). Default `all` = GPG when present
  + at least one integrity hash mandatory. Security checks stop the run, never
  warn-and-continue. The official repo gets NO exemption — `official` only
  affects install-new scope and ChangeLog tracking. GPG keys are pinned on first
  use (TOFU): a changed fingerprint is refused as key substitution. A repo that
  fails vetting is auto-quarantined (markers under CACHE_DIR; see repo.rs) and
  provides no packages until trust-repo/vet-repo clears it.

- ERRORS ARE STRINGS by design. Keep messages short, specific, and actionable
  (the user sees them prefixed with "slacker: error:").

- TESTS: `cargo test` needs no root and no network (they use in-memory/temp
  fixtures). Add a unit test next to any matching/priority change — that family
  of tests is the project's safety net. Live bare-metal testing still catches
  things sandboxes miss (encoding, paging, arch edge cases): build debug,
  reinstall, test on a real -current box.

- edition 2021 on purpose (broad Rust compatibility). The crate's own edition is
  2021, but the EFFECTIVE MSRV is 1.85.1: a transitive dep (clap_lex 1.1.0) is
  edition 2024, which only stabilised in Rust 1.85. `cargo msrv find` confirms
  this; declare it as `rust-version` and freeze deps (Cargo.lock + cargo vendor)
  before shipping to a distro. Direct deps: clap, ureq (rustls TLS — no system
  OpenSSL), md-5, and regex (for blacklist rule matching); think hard before
  adding another. TLS is rustls with bundled roots on purpose: it drops the
  libssl dependency so the binary runs on an older base than it was built on.
